# 任务交接文档 Skill

## 概述

在项目根目录维护一份结构化的 `progress.md`，让用户可以在任意新对话中快速恢复任务上下文，最小化 token 占用。

---

## 文档模板（6 模块固定结构）

```
# [项目名] — 任务交接文档

> 最后更新：YYYY-MM-DD HH:MM:SS
> 维护约定：阶段性产出完成后更新；新对话开始时先读取本文档

---

## 1. 任务概览
- 当前目标：一句话
- 项目背景：一段话
- 当前优先级：🔴高 / 🟡中 / 🟢低，最多列 3 项

## 2. 进度状态
- ✅ 已完成
- 🔄 进行中
- ⏳ 待办

## 3. 交付物清单
分三类表格：
1. workspace 文件（路径、用途、状态）
2. tdrive/资产库文档（名称、主要内容、状态）
3. 本地参考文档（非交付物）

状态标记：✅完成 / 🔄进行中 / ⏳待开始 / ❌已废弃

## 4. 关键决策
已确认的技术方案、重要约定、排除的选项。
每项一行标题 + 一句说明，不展开论述。

## 5. 核心上下文
必要的文件路径、关键命名、数据口径、用户角色、工具约定等。
只列事实，不臆想。不确定的标注"待确认"。

## 6. 下一步行动
- **新对话启动指令**：可直接复制的 prompt 文本
- **当前可执行任务**：无需等待决策的事项
- **待用户决策**：需要用户拍板的事项
```

---

## 执行流程

### 1. 读取现状
- 读取已有 `progress.md`（如果存在）
- 列出当前项目文件结构
- 如有 tdrive，列出资产库文档
- 如有 git，查看最近 commit

### 2. 对比变化
- 识别新增/删除/修改的文件
- 识别已完成的任务（从 🔄 变为 ✅）
- 识别新增的待办

### 3. 更新文档
- **先获取当前系统时间**：执行 `date "+%Y-%m-%d %H:%M:%S"`，用结果更新 `最后更新` 字段。禁止跳过这一步直接填日期或时间。
- 只更新有变化的模块
- 不重写全文
- 保持每个条目一句话，精炼

### 4. 提醒同步
- 完成后提醒用户：是否同步到 tdrive

---

## 铁律

1. **只写已经确认过的事实**。不确定的标注"待确认"或留空。
2. **不带倾向**。待决策事项只列出选项，不暗示推荐。
3. **不臆想数据**。没见过的文件名、路径、数据口径一律不编造。
4. **保持精炼**。每个条目控制在一句话。这是交接文档，不是周报。
5. **不自行推送 GitHub**。原型或文档更新后，只在用户明确说「推送」「push」「推到 GitHub」时才执行 git push。GitHub 上必须是用户验证过的版本。
6. **更新时间取系统时间，绝不推断**。`最后更新` 日期时间必须严格取当前系统时间（执行 `date "+%Y-%m-%d %H:%M:%S"`），不允许根据上下文推断、不允许沿用上次日期、不允许凭感觉填。"推断"出的时间即便碰巧正确也是错的——因为流程不可靠。

---

## 触发时机

- 用户主动说"更新 progress" / "更新交接文档" / "同步进度"
- 每完成一个阶段性产出后，**主动询问**用户是否更新
- 不要每句对话后都更新

---

## 输出格式

更新完成后简洁告知：
```
progress.md 已更新：
- [模块名]：变化内容简述
- [模块名]：变化内容简述
是否需要同步到 tdrive？
```
